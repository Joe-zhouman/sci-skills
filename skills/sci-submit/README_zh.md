# sci-submit

投稿战役管理器。不是"帮你写一封投稿信"的工具——是"陪你把一篇文章投了十几次还没放弃"的工具。

## 为什么有这个技能

我投过一篇文章，一年半，换了十几个期刊。

每一次被拒之后，打开下一个投稿系统，重新填一遍：标题、摘要、作者、单位、基金号、关键词、推荐审稿人、回避审稿人……然后重新写 cover letter。明明 90% 的内容跟上次一模一样，但期刊名字换了，又得从头改。

最惨的不是被拒。最惨的是每次被拒之后，导师说"没事，下一个"——他不知道填一次投稿系统要四十分钟。他不知道你说不出口的那些话：文章水平可能不够投 Nature 子刊、这个期刊审稿周期比你毕业时间还长、学校人事处根本不认这个出版社。

你不能跟导师说"我们水平不够"。这不 work。在等级里，你说"水平不够"等于说你懒、你没信心、你不努力。你也不能跟大老板说"你定的目标是错的"。你就是不能。

所以有了这个技能。它解决三件事：
1. **元数据写一次，处处复用**——`manuscript-meta.md` 存在你项目里。关键是它的设计：不是给你读的，是给你贴的。每一个投稿系统的 input 框对应文件里一个代码块。好记的（作者名、标题）不设块——不好记的（邮箱、ORCID、基金号、邮编）一人一块、一号一块。打开文件，三击全选 → Ctrl+C → 切到浏览器 → Ctrl+V。填一个投稿系统从四十分钟降到五分钟。然后这个文件留在那里，下一次被拒换刊，打开再贴一遍，不用重填。
2. **投稿战役可追踪**——`submit-history.md` 记录每一次投稿，`journal-shortlist.md` 管理候选期刊
3. **硬约束下的务实路径**——不画饼，不硬刚，用数据说话

## 七大工作流

| 场景 | 工作流 |
|---|---|
| 首次使用，整理投稿元数据 | **A: Metadata Setup** |
| 选刊、比较期刊、查分区 | **B: Journal Selection** |
| 投稿 / 过投稿系统（封面信 + 逐页引导） | **C: Submission** |
| 被拒了，换期刊 | **D: Rejection & Switching** |
| 投稿后状态追踪、催稿 | **E: Post-submission Tracking** |
| 校对 proof / 校样 | **G: Proof Review** |
| 导师非要投 XX、评职称硬指标 | **F: Hard Constraints & Advisor Mgmt** |

## 数据源

- **EasyScholar API**：实时查询 SCI 分区、中科院分区、影响因子、预警名单
- **中国科协分级目录**：59 个领域的 T1/T2/T3 分级，离线可查

## 项目文件结构

```
project/
├── manuscript.tex
├── assets/                     # 可复用投稿素材
│   ├── declarations.tex        # 标准声明（冲突利益、作者贡献等）
│   └── author-bios.tex         # 作者简介
└── sci-skills/sci-submit/                 # 决策文件
    ├── manuscript-meta.md      # 可复用投稿元数据 + 封面信 Cheat Sheet
    ├── hard-constraints.md     # 硬性约束
    ├── submit-history.md       # 投稿时间线
    ├── journal-shortlist.md    # 候选期刊排名 + 每刊的封面信论文素材
    └── cover-letter/
        └── <journal-name>-<YYYY-MM-DD>/
            └── cover-letter.tex
```

## 人在环路中

这个技能是参谋，不是代笔。它组织信息、查数据、拦截常见错误、记录决策——但人来拍板。你比任何模型都更懂你的领域、你的导师、你的处境。

## 反馈与贡献

如果你也经历过反复投稿的痛苦，欢迎告诉我你的场景和痛点——这个技能会持续进化，因为它的需求来自真实的伤口。
